# PRD: Conductor Phase 2 — Unified Session Model with Rehydrate/Dehydrate

## Introduction

Currently claude-code-slack treats Slack sessions and Zellij sessions as separate entities with distinct UI flows, data models, and lifecycles. This creates friction: a conversation started in Slack can't be continued in a terminal, and terminal work can't be shared back to Slack.

Phase 2 unifies sessions into a single model where any session can flow between Slack and Zellij. **Rehydrate** converts a Slack-only session into a live terminal session (using `claude --resume` to carry forward context). **Dehydrate** posts a Zellij session's history to a Slack DM thread, enabling continued conversation via Slack replies. The UI merges into one detail view that shows both Slack metadata and Zellij controls when applicable.

## Goals

- Enable seamless workflow continuity: start a conversation in Slack, pick it up in a Zellij terminal, and vice versa
- Unify the session data model so one session record can have both a Slack thread and a Zellij process
- Remove the Slack/Zellij type distinction from the UI — sessions are just "sessions" with different capabilities attached
- Support bidirectional sync: Slack messages appear in Zellij context, Zellij output can be posted to Slack

## User Stories

### US-001: Unified session data model
**Description:** As a developer, I need a single session record that can hold both Slack and Zellij associations so that sessions can flow between the two mediums.

**Acceptance Criteria:**
- [ ] Session table supports both `thread_ts`/`channel_id` (Slack) and `zellij_session_name` (Zellij) on the same row
- [ ] Existing Slack-only and Zellij-only sessions migrate cleanly (backward compatible)
- [ ] `session_type` column is removed or repurposed — a session's capabilities are derived from which fields are populated
- [ ] Primary key strategy handles both: Slack sessions keyed by `thread_ts`, Zellij by `zellij:<name>`, unified sessions by whichever came first
- [ ] `uv run pytest` passes with updated store tests
- [ ] Typecheck/lint passes

### US-002: Rehydrate — Resume Slack session in Zellij
**Description:** As a user, I want to tap "Zellij" on a Slack-only session to open it in a live terminal, resuming the Claude conversation from where Slack left off.

**Acceptance Criteria:**
- [ ] Slack-only sessions in the web UI show a "Zellij" button (replaces old "Resume" for this context)
- [ ] Clicking "Zellij" creates a Zellij background session running `claude --resume <session_id> --worktree <name>`
- [ ] The session record is updated to also have a `zellij_session_name`
- [ ] After rehydration, the detail view shows both the Slack thread link and Zellij Attach/Resume controls
- [ ] The "Zellij" button is disabled/hidden if there's already an active Zellij session for this session
- [ ] Typecheck/lint passes

### US-003: Dehydrate — Post Zellij session to Slack
**Description:** As a user, I want to tap "Slack" on a Zellij-only session (with no active Zellij process) to create a Slack DM thread containing a summary of what was done, enabling continued conversation via Slack.

**Acceptance Criteria:**
- [ ] Zellij-only sessions with no active Zellij process show a "Slack" button in the detail view
- [ ] Clicking "Slack" posts a message to the configured DM channel summarizing the Zellij session's work
- [ ] The summary is generated by running `claude --resume <session_id> -p "Summarize what was accomplished in this session in a few bullet points"`
- [ ] If summary generation fails, fall back to a simple "Session started" anchor message
- [ ] A Slack thread is created and the session record is updated with `thread_ts` and `channel_id`
- [ ] Subsequent Slack replies in that thread continue the Claude conversation (existing Slack handler picks it up)
- [ ] The "Slack" button is disabled/hidden if the session already has a Slack thread
- [ ] Typecheck/lint passes

### US-004: Unified detail view
**Description:** As a user, I want a single session detail view that shows all available controls (Slack link, Zellij attach/resume, dehydrate/rehydrate) based on what's associated with the session.

**Acceptance Criteria:**
- [ ] Remove the separate Slack detail view and Zellij detail view — replace with one unified component
- [ ] If session has Slack: show Slack thread URL link, date, channel info
- [ ] If session has Zellij and it's alive: show "Attach" button and status indicator
- [ ] If session has Zellij and it's stopped: show "Resume" button
- [ ] If session has Slack but no Zellij: show "Zellij" (rehydrate) button
- [ ] If session has Zellij but no Slack: show "Slack" (dehydrate) button (only when Zellij is stopped)
- [ ] Title editing works on the unified view
- [ ] Delete button handles cleanup of both Zellij process and DB record
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-005: Remove type badges and unify sidebar
**Description:** As a user, I want the sidebar to show sessions without Slack/Zellij type differentiation since sessions are now unified.

**Acceptance Criteria:**
- [ ] Remove "S" and "Z" type badges from sidebar session items
- [ ] Replace with capability indicators: small icons for Slack (speech bubble) and Zellij (terminal) that show which associations exist
- [ ] Both icons shown for sessions that have both associations
- [ ] Alive dot indicator still shown for sessions with an active Zellij process
- [ ] Session grouping by project remains unchanged
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-006: Bidirectional sync — Slack messages forwarded to Zellij
**Description:** As a user, when I reply in a Slack thread that has an active Zellij session, I want my message to also reach the Claude session in the terminal.

**Acceptance Criteria:**
- [ ] When a Slack message arrives for a session that has an active Zellij process, the Slack handler detects this
- [ ] Instead of (or in addition to) running a separate `claude -p` subprocess, the message content is written into the Zellij session via `zellij action write-chars`
- [ ] Claude's response in the terminal is captured and posted back to the Slack thread
- [ ] If the Zellij session is not alive, fall back to the existing Slack-only subprocess flow
- [ ] Typecheck/lint passes

### US-007: Bidirectional sync — Zellij output posted to Slack
**Description:** As a user, when Claude produces output in a Zellij terminal session that has a linked Slack thread, I want key outputs posted to the Slack thread.

**Acceptance Criteria:**
- [ ] When a Zellij session with a linked Slack thread produces a response, the response is posted to the Slack thread
- [ ] Output is captured by monitoring the PTY output during active WebSocket sessions, or by polling session state
- [ ] Only meaningful Claude responses are posted (not raw terminal noise — filter for structured output)
- [ ] Posts are throttled/batched to avoid flooding Slack (e.g., at most one post per 30 seconds)
- [ ] If the Slack thread doesn't exist yet, this is a no-op (don't auto-create threads)
- [ ] Typecheck/lint passes

### US-008: Backend API updates
**Description:** As a developer, I need updated API endpoints to support rehydrate, dehydrate, and the unified model.

**Acceptance Criteria:**
- [ ] `POST /api/sessions/{key}/rehydrate` — creates Zellij session for a Slack-only session, updates DB record
- [ ] `POST /api/sessions/{key}/dehydrate` — generates summary, posts to Slack, updates DB record with thread_ts
- [ ] `GET /api/sessions` returns unified session objects with both Slack and Zellij fields (nullable)
- [ ] `GET /api/sessions` response includes `has_slack: bool` and `has_zellij: bool` convenience fields
- [ ] Existing Zellij CRUD endpoints continue to work
- [ ] Typecheck/lint passes

## Functional Requirements

- FR-1: Merge `session_type` into derived state — a session "has Slack" if `thread_ts` is set, "has Zellij" if `zellij_session_name` is set
- FR-2: `POST /api/sessions/{key}/rehydrate` must run `claude --resume <session_id> --worktree <name>` in a new Zellij background session
- FR-3: `POST /api/sessions/{key}/dehydrate` must generate a session summary via Claude CLI and post it as a Slack message to the DM channel, creating a thread
- FR-4: If summary generation fails (timeout, error), dehydrate must fall back to posting a simple anchor message ("Session dehydrated from Zellij terminal")
- FR-5: The Slack message handler (`slack_app.py`) must check if an incoming thread's session has an active Zellij process and route accordingly
- FR-6: Bidirectional sync from Slack→Zellij uses `zellij action write-chars` to inject the user's message
- FR-7: Bidirectional sync from Zellij→Slack monitors terminal output and posts formatted responses to the linked Slack thread
- FR-8: Output posting to Slack must be throttled (max 1 post per 30 seconds) and filtered to exclude raw terminal control sequences
- FR-9: The unified detail view must dynamically show/hide controls based on which associations (Slack/Zellij) the session has
- FR-10: Session deletion must clean up both Zellij process (if alive) and Slack association (no Slack cleanup needed — threads persist)

## Non-Goals

- No real-time terminal streaming to Slack (batch/summary only)
- No multi-user collaboration on the same session
- No automatic rehydration/dehydration — always user-initiated via button
- No changes to the `claude` CLI itself — only use existing `--resume`, `-p`, and `--worktree` flags
- No Slack slash commands for rehydrate/dehydrate — web UI only for Phase 2
- No session merging (combining two separate sessions into one)

## Design Considerations

- The unified detail view should feel like one cohesive panel, not two views stitched together. Slack metadata (thread link, date) at the top, then action buttons below, then terminal (when attached) taking remaining space.
- Capability icons in the sidebar (speech bubble for Slack, terminal icon for Zellij) should be subtle — small, monochrome, next to the session title. Both appear when a session has both associations.
- The "Zellij" and "Slack" action buttons should use distinct colors to signal the direction of conversion (e.g., green for Zellij/terminal, purple for Slack).

## Technical Considerations

- **Schema migration**: Adding `zellij_session_name` as a nullable column to existing sessions. The current key format (`thread_ts` for Slack, `zellij:<name>` for Zellij) should be preserved as the primary key — the new column is an association, not a key change.
- **Summary generation**: Dehydrate calls `claude --resume <session_id> -p "Summarize..."` as a subprocess. This may take 10-30 seconds. The API should return immediately with a "dehydrating" status and update asynchronously, or use a loading state on the frontend.
- **Slack→Zellij message injection**: `zellij action write-chars` sends raw keystrokes. Need to handle multi-line messages carefully (escape newlines or use a paste-friendly format).
- **Zellij→Slack output capture**: This is the hardest part. Options: (a) parse PTY output for Claude's structured response markers, (b) use `claude --resume -p` to ask for a summary periodically, (c) monitor the session's conversation log file if Claude writes one. Option (a) is fragile; (b) is simpler but adds latency; (c) depends on Claude CLI internals.
- **Race conditions**: A session could receive a Slack message while the Zellij terminal is actively being used. Need to queue or warn rather than inject simultaneously.
- **Existing Zellij endpoints** (`POST /api/sessions/zellij`, `DELETE /api/sessions/zellij/{key}`, etc.) should remain functional for backward compatibility but may be refactored to use the unified model internally.

## Success Metrics

- A user can start a conversation in Slack, rehydrate to Zellij, work in the terminal, dehydrate back to Slack, and continue in Slack — all on the same session
- Slack messages sent to a thread with an active Zellij session reach the terminal within 5 seconds
- Zellij responses appear in the linked Slack thread within 60 seconds
- No regression in existing Slack-only or Zellij-only workflows

## Open Questions

- **Zellij→Slack output capture strategy**: What is the most reliable way to detect and extract Claude's responses from PTY output? Should we use structured output markers, periodic summarization, or conversation log files?
- **Dehydrate async UX**: Should the dehydrate button show a loading spinner and block until the summary is posted, or return immediately and show "Posting to Slack..." status?
- **Session key after unification**: If a Slack session (keyed by `thread_ts`) gets rehydrated, does the key stay as `thread_ts`? Or should we migrate to a UUID-based key scheme?
- **Conflict resolution**: If both Slack and Zellij are active simultaneously and both send a message, which takes priority? Queue? First-come-first-served?
- **DM channel configuration**: Should dehydrate always post to a single configured channel (`SLACK_CRON_CHANNEL`), or should the user pick a channel? Or use the user's DM channel?
